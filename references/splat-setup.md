# Splat Project Setup

Read this when initializing a new matching decomp project from a baserom.

## Initialize

Ask whether to initialize a git repository.

Use `uv` (not pip) for version locking:

```bash
uv init --bare
uv add "splat64[mips]"
```

Copy ROM as `baserom.<extension>` (keep original extension):

```bash
cp "/path/to/Game Name.n64" baserom.n64
```

Generate config (splat byteswaps internally):

```bash
uv run -m splat create_config baserom.n64
```

After config generation, splat creates byteswapped `baserom.z64`. **Always use `baserom.z64` from this point** — the build expects z64 format.

Enable undefined symbol paths in the yaml (uncomment):

```yaml
undefined_funcs_auto_path: undefined_funcs_auto.txt
undefined_syms_auto_path: undefined_syms_auto.txt
```

Do **not** enable `hardware_regs` or `libultra_symbols` — they cause symbol mismatches and unnecessary work.

Split the ROM:

```bash
uv run -m splat split <game>.yaml
```

Splat generates `include/macro.inc` automatically — do not create it manually.

## .gitignore

Avoid committing copyrighted ROMs:

- `baserom.z64` (or input ROM name)
- Generated folders: `asm/`, `bin/`, `assets/`, etc.

## asm/ Folder Rule

`asm/` is generated by splat and **not** checked in. Never edit asm files to fix build issues — changes are lost. Fix YAML config or symbol files instead.

Splat suggests file splits in output (object boundaries via zero-padding between 0x10-aligned functions). Add those to yaml **after** a basic matching build works.

Split detection: splat detects boundaries when a function ends at e.g. 0x1C and the next starts at 0x20 with padding. Without padding, a boundary may still exist but splat cannot detect it.

## Subsegment Format

```yaml
subsegments:
  - [0x1050, asm]                    # name defaults to "1050"
  - [0xA0CD0, asm, libultra/A0CD0]   # explicit name → libultra/ folder
```

Add a third element to rename. Names with slashes become folders.
